### [F-2] [HIGH] [BUG] A shipped migration uses `CREATE INDEX CONCURRENTLY`, which is incompatible with `golang-migrate` transactions

- **Affected feature:** DB migrations
- **Location:** `migrations/20260323000002_perf_indexes.up.sql:4-18`, `docs/ARCHITECTURE.md:82`
- **Evidence:**
  ```sql
  -- Enable pg_trgm for trigram indexes on SIM search fields
  CREATE EXTENSION IF NOT EXISTS pg_trgm;

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_sims_tenant_created
      ON sims (tenant_id, created_at DESC, id DESC);

  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_sims_tenant_iccid_trgm
      ON sims USING gin (iccid gin_trgm_ops);
  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_sims_tenant_imsi_trgm
      ON sims USING gin (imsi gin_trgm_ops);
  ```
  ```md
  | Migration | golang-migrate | Latest | Versioned, reversible SQL migrations |
  ```
- **Issue:** PostgreSQL rejects `CREATE INDEX CONCURRENTLY` inside a transaction, while the architecture explicitly standardizes on `golang-migrate`, which wraps SQL migrations transactionally. That makes this migration fail on a clean apply path, matching the Phase 10 carry-over about fresh deploy migration breakage. Because this is already in the shipped migration chain, new environments can fail before the app becomes usable.
- **Recommended fix:** Replace the concurrent index statements in the transactional migration with non-concurrent variants, or split them into a migration strategy that is explicitly executed outside `golang-migrate` transactions and documented as such.
- **Suggested amil action:** `bugfix`
- **Estimated effort:** `M` (30min-2h)
