## Bug Patterns & Prevention Rules

- [2026-04-12] PAT-001 [STORY-059]: **BR/acceptance tests assert behavior, not implementation — update BR tests when acceptance criteria change the rule.** — Root Cause: When STORY-059 AC-6 changed `stolen_lost` from absorbing terminal state to `stolen_lost → terminated` allowed, `internal/store/sim_test.go` was updated but `internal/store/sim_br_test.go` (BR-1 assertions: `TestBR1_StolenLostHasNoOutboundTransitions`, `TestBR1_StolenLostCannotTerminate`) still asserted the OLD rule and failed CI. — Prevention: Planner MUST grep for `*_br_test.go` files touching any state/rule being changed by an AC and list them in task Files/Context refs. Developer MUST grep for BR-NNN references for every business rule modified and update assertions in the same task. Gate verifies by running the full test suite across all packages, not just the files listed as modified. — Affected: Go test layer, BR-assertion tests, anywhere behavior-contract tests exist separately from unit tests.
- [2026-04-12] PAT-002 [STORY-059]: **Duplicated utility functions drift — the #1 example is `extractIP` pattern repeated across packages.** — Root Cause: AC-1 fixed the IPv6-unsafe `split-on-last-colon` `extractIP` in `internal/analytics/anomaly/detector.go`, but an identical buggy pattern exists in `internal/gateway/bruteforce.go:91` (found by Gate, documented as out-of-scope for STORY-059). — Prevention: When fixing a utility function, grep for other occurrences of the same pattern across the codebase. Plan task should enumerate sibling locations. Long-term: extract shared helper into `internal/net/addr.go` or similar. — Affected: Utility code, networking helpers, anything that parses `host:port` or performs input sanitization.
- [2026-04-12] PAT-003 [STORY-060]: **AT_MAC field MUST be zeroed before computing HMAC-SHA1 in EAP-SIM/AKA (RFC 4186 §10.15).** — Root Cause: Original `verifyMAC` implementation in `internal/aaa/eap/sim.go` computed HMAC over the raw packet including the AT_MAC value bytes, rather than zeroing them first. This caused HMAC mismatches with compliant supplicants and was masked by the now-removed dual-MAC test-compat path (`verifySimpleSRES`). — Prevention: Any HMAC/MAC computation over a packet that contains the MAC field itself MUST zero the MAC bytes in the computation buffer before hashing (RFC pattern). Never use the raw packet bytes. Create a copy, zero the attribute payload, then HMAC the copy. Also see VAL-007 → DEV-149. — Affected: EAP-SIM (`sim.go`), EAP-AKA (`aka.go`), any future EAP method with an inline AT_MAC attribute.
- [2026-04-17] PAT-005 [STORY-083 F-B1]: **Write-before-close-channel — in a goroutine network with a shared done channel, broadcast AFTER the last wire operation, not before.** — Root Cause: `Peer.Close()` called `close(closeCh)` before writing the DPR. The watchdog exited on `<-closeCh`, `Run` called `conn.Close()`, and the subsequent `conn.Write(dpr)` hit a closed connection. Under the race detector this ordering occurred ~15% of the time. — Prevention: (1) Close broadcast channels AFTER the last operation they must not abort; (2) gate the broadcast behind the "point of no return" (final write done or provably errored); (3) where idempotency is available (`net.Conn.Close`), use it defensively at the tail of `Run`. — Affected: Any shutdown path that (a) sends a final protocol frame and (b) uses `close(ch)` to unblock sibling goroutines. Verified by 50/50 `-race -count=50` on `TestPeerGracefulClose`.
- [2026-04-17] PAT-006 [STORY-083 F-A3]: **Single-writer metric classification — counters with an enum `reason` label must be incremented by exactly one layer.** — Root Cause: `DiameterSessionAbortedTotal{reason}` was incremented by both `Client` (on `ErrPeerNotOpen`) and `Engine` (on any `OpenSession` error), yielding double-counts and two reason values set for one real event. — Prevention: (1) Identify the "owner" layer — typically the highest that can distinguish all enum values; (2) lower layers return typed sentinels (`errors.Is`-compatible) instead of incrementing counters; (3) tests assert `sum(counter{all_reason_values}) == 1` for one logical event, not just individual label assertions, to catch double-counts. — Affected: Any multi-layer architecture where error classification labels are shared between call site and callee (e.g., `reason=peer_down|ccr_i_failed|timeout|reject`).
- [2026-04-17] PAT-004 [STORY-086]: **Never declare a FK `REFERENCES parent(id)` against a LIST/RANGE-partitioned table unless the referenced column set exactly matches a UNIQUE or PRIMARY KEY constraint on that partitioned table.** — Root Cause: `20260413000001_story_069_schema.up.sql:144` declared `sim_id UUID NOT NULL REFERENCES sims(id)`. The `sims` table has been partitioned by LIST since day 1 (`20260320000002_core_schema.up.sql`) with composite PK `(id, operator_id)`. PostgreSQL rejects FK references to partitioned tables unless the FK columns match the full unique/PK column set — `id` alone is not sufficient. This caused a silent DDL failure: `ERROR: there is no unique constraint matching given keys for referenced table "sims"`. The `CREATE TABLE` never committed; the six sibling tables in the same migration committed because they do not reference `sims`. — Prevention: Before writing `REFERENCES sims(id)` (or any partitioned table), check whether the PK is composite. If the referenced table is partitioned: (a) add a `UNIQUE(id)` covering index on the parent (if data model allows), OR (b) use a `check_X_exists` BEFORE trigger instead of a FK constraint (STORY-064/DEV-169 precedent: `esim_profiles`, `ip_addresses`, `ota_commands`, `sms_outbound`). A migration that fails silently due to this bug leaves `schema_migrations` version advanced (if the transaction wrapped all DDL at file level) or partially committed (if not wrapped). A boot-time `schemacheck.Verify` guard (`internal/store/schemacheck`) catches this before the server accepts traffic. — Affected: any migration that adds a table with a FK reference to `sims`, `sessions`, `cdrs`, `sim_state_history`, `operator_health_logs`, or any future ListRange-partitioned table.
