- Global competitors mapped: Enea AAA, Alepo, Nokia, Radiator, Aradial, emnify, 1NCE, MAVOCO, Stacuity

- 10M+ SIM cards to manage
- 3-4 different mobile operators (Turkcell, Vodafone, TT Mobile + potential MVNO)
- Private APN, operator-managed APN, customer-managed APN support
- Multi-operator intelligent routing
- Policy enforcement engine (mini-PCRF)
- Real-time analytics
- Custom AAA core in Go (not FreeRADIUS)

## Formal Discovery Q&A

### Q1: Target Customer Profile
**Question:** Who is the primary target customer? (A) Mobile operators, (B) Enterprise customers for IoT/M2M fleet management, (C) MVNO/MVNEs, (D) We offer as managed SaaS/platform, (E) Multiple segments
**Answer:** Primarily B (Enterprise customers) but D (managed service/SaaS) support is also required.
**Decision:** Primary target = Enterprise (logistics, energy, healthcare, etc.) managing their own IoT/M2M fleets. Deployment model must support both on-premise (for enterprise) and SaaS/managed service. Multi-tenant architecture required for SaaS model.
**Notes:** This means the platform needs: tenant isolation, self-service portal for enterprises, and a managed-service operational layer for SaaS offering.

### Q2: Enterprise Customer Verticals
**Question:** Which industry verticals are targeted? (A) Logistics/Fleet, (B) Energy/Utilities, (C) Healthcare, (D) Fintech/POS, (E) Smart city, (F) Agriculture, (G) Sector-agnostic
**Answer:** Primarily B (Energy/Utilities) but must think sector-agnostic (G).
**Decision:** Platform must be sector-agnostic by design. Energy/Utilities is the beachhead market (smart meters, SCADA) but architecture must not be coupled to any specific vertical. No vertical-specific features in core — vertical needs handled via policy configuration and API integrations.
**Notes:** Energy/Utilities first makes sense: high SIM density (millions of smart meters per utility), long device lifecycles, strict security/compliance requirements, predictable traffic patterns. Good anchor vertical for 10M+ SIM target.

### Q3: Business Model
**Question:** Revenue model? (A) Per-SIM monthly, (B) Platform license, (C) Hybrid base + per-SIM, (D) Usage-based, (E) Decide later
**Answer:** E — not yet decided, will determine later.
**Decision:** Deferred to product definition phase. Architecture should support metering/usage tracking to enable any pricing model later.
**Notes:** Platform must have built-in usage metering regardless of final pricing model — this is an architectural requirement even if business model is deferred to post-v1.

### Q4: Operator Integration Model
**Question:** Technical integration status with operators (Turkcell, Vodafone, TT)? (A) Agreements exist, private APNs defined, (B) Relationship exists but no technical integration, (C) From scratch, (D) One operator exists, others later
**Answer:** C — from scratch, operator relationships need to be established too.
**Decision:** Operator integration is a business development dependency. Platform must be developed operator-agnostic with a pluggable adapter pattern so any operator can be onboarded independently. MVP can work with simulated/mock operator interfaces initially.
**Notes:** This is a critical path risk. Operator agreements take months. Development should NOT be blocked by operator negotiations. Mock/simulator layer is essential for development and demo purposes.

### Q5: Project Type & Platform (detected)
**Question:** How will the product be delivered? (A) Fullstack — web portal + AAA engine, single repo, docker-compose, (B) API-backend only, (C) Fullstack separate repos, (D) Backend core + minimal admin dashboard single repo
**Answer:** A — Fullstack, web management portal + backend AAA engine, single repo, docker-compose.
**Decision:** Project type = fullstack. Monorepo with web management portal (frontend) + AAA backend engine. Containerized via docker-compose. Single deployable unit.
**Notes:** Fullstack monorepo keeps deployment simple for on-premise enterprise customers. Docker-compose for dev/small deployments, Kubernetes for production/SaaS scale.

### Q6: MVP Scope
**Question:** Should we phase features into MVP/v2/v3, or ship all 5 layers as a single full-featured MVP?
**Answer:** All layers (L1-L5) must be in MVP. Full product from day one. Must compete with global rivals (Enea, Alepo, Nokia) and decisively surpass Turkish alternatives (Mobile ITM SCM).
**Decision:** No phased feature delivery. MVP = full-featured product across all 5 layers: AAA Core, SIM/APN Lifecycle, Multi-Operator Orchestration, Policy Engine (PCRF-lite), and BI/Analytics. This is a competitive positioning decision — entering the market with a partial product won't work against established players.
**Notes:** This significantly increases development scope. Development phases will be about building layers bottom-up (L1→L5) but ALL ship together as v1. No public release until all layers are functional.

### Q7: Team & Development Capacity
**Question:** Team size for development? (A) Solo + Claude Code, (B) 2-5 people, (C) 5-15, (D) 15+, (E) Solo now, team later
**Answer:** A — Solo developer + Claude Code (Amil). All development will be done with Claude Code.
**Decision:** Solo developer with AI-assisted development (Claude Code / Amil). This fundamentally shapes architecture decisions: must maximize code generation efficiency, prefer convention over configuration, use well-known frameworks with strong ecosystem, minimize custom infrastructure that needs manual debugging.
**Notes:** Critical implications:
- Go backend + React/Next.js frontend (both have excellent Claude Code support)
- Heavy use of code generation, OpenAPI specs → auto-generated clients
- Docker-compose for local dev, no complex local setup
- Automated testing is non-negotiable (solo dev = no manual QA team)
- Stories must be small, self-contained, independently testable
- AUTOPILOT mode will be heavily used for story execution

### Q8: Tech Stack Preference
**Question:** Frontend framework preference? (A) React + Next.js, (B) React + Vite, (C) Vue + Nuxt, (D) Svelte + SvelteKit, (E) No preference, most efficient one
**Answer:** E — no preference, most efficient. Backend confirmed Go (performance is critical).
**Decision:**
- Backend: Go (confirmed, performance-critical AAA engine)
- Frontend: React + Vite (SPA) — rationale below
**Notes:** Why React + Vite over Next.js:
- This is an admin/management portal, not a public website → no SEO needed → no SSR needed
- Vite = faster dev builds, simpler deployment (static files behind Go API or Nginx)
- React has the largest component ecosystem (charts, tables, data grids for 10M+ SIM views)
- Claude Code has excellent React generation capabilities
- SPA fits the "single docker-compose" delivery model (Go serves static + API)
- Alternative considered: Next.js adds SSR complexity we don't need for a B2B management portal

### Q9: Security & Compliance Requirements
**Question:** Which security/compliance requirements must be in v1? (A) BTK, (B) KVKK, (C) GDPR, (D) ISO 27001 audit logging, (E) RadSec, (F) RBAC, (G) All
**Answer:** G — All of them.
**Decision:** Full security & compliance suite in v1: BTK, KVKK, GDPR, ISO 27001 audit logging, RadSec (RADIUS/TLS), RBAC.
**Notes:** This aligns with the "compete globally" decision. Enterprise customers (especially energy/utilities) require compliance certifications before procurement. Audit logging and RBAC are table stakes. GDPR signals international ambition beyond Turkey.

### Q10: Database & Data Layer
**Question:** Proposed data layer: PostgreSQL (config/SIM), Redis (session cache), TimescaleDB (analytics), append-only PG partitions (audit), NATS (event bus). Approve or modify?
**Answer:** PG, TimescaleDB, NATS — confirmed. (Redis implied as part of standard stack.)
**Decision:** Data layer stack confirmed:
- PostgreSQL: primary relational store (SIM inventory, config, APN, operators, tenants)
- Redis: session cache, hot data, real-time counters
- TimescaleDB: time-series analytics, accounting data, usage metrics (PG extension, same cluster)
- NATS: event bus / message broker (lightweight, Go-native)
- Audit: append-only PG partitions + S3 archive for long-term retention
**Notes:** Key advantage: PG + TimescaleDB = single database engine for both OLTP and analytics. Reduces operational complexity for solo dev. NATS over Kafka = much simpler to operate, sufficient for our event patterns.

### Q11: eSIM / eUICC Support Depth
**Question:** eSIM integration level? (A) Inventory only, (B) SM-DP+ integration, (C) Own SM-DP+ server, (D) SGP.22+SGP.32
**Discussion:** User asked for detailed comparison of all levels + Turkey regulatory feasibility. Research revealed:
- BTK 2025 regulation: permanent roaming banned, data localization required, eSIM must use local operators
- Own SM-DP+ (C) requires GSMA SAS-SM certification, FIPS 140-2 L3 HSM — unrealistic for solo dev
- SGP.32 ecosystem still immature (released 2023)
**Answer:** Approved A+B approach.
**Decision:** eSIM strategy = Level A (inventory/tracking) + Level B (SM-DP+ API integration with operator's infrastructure). Operator handles provisioning, Argus provides unified management portal. SGP.32 eIM support deferred to future. Own SM-DP+ hosting rejected permanently.
**Notes:** BTK compliance is critical — all eSIM operations must go through licensed Turkish operators. This is actually an advantage: positions Argus as the management layer operators need to offer their enterprise customers.

### Q11b: eSIM Strategic Positioning
**Question:** Should eSIM be a first-class citizen in the product (front-and-center in UX, unified with physical SIM, cross-operator profile switch, bulk provisioning, policy binding)?
**Answer:** Yes — eSIM as first-class citizen confirmed.
**Decision:** eSIM is not a secondary feature but a core differentiator. Unified SIM view (physical + eSIM), cross-operator profile switching, bulk eSIM provisioning, eSIM+policy binding, eSIM analytics, BTK compliance dashboard. Argus positions itself as orchestration layer ON TOP of operators' SM-DP+ — not a competitor to operators but an enabler.
**Notes:** Key competitive positioning:
- vs Global: "Multi-operator eSIM management"
- vs TR local: "3 operators' eSIM from single portal"
- vs Operators: "We use YOUR SM-DP+, we're your enterprise channel"
This is a GTM advantage — operators may want to resell/white-label Argus to their enterprise customers.

### Q12: Deployment Environment
**Question:** Deployment model? (A) On-prem only, (B) Cloud SaaS only, (C) Both — same artifact, (D) Cloud first, on-prem later
**Answer:** C — both, same artifact must work on-prem and cloud.
**Decision:** Single artifact, dual deployment. Docker-compose for on-prem, Kubernetes/Helm for cloud SaaS. All config via environment variables, no cloud-specific dependencies baked in. S3-compatible storage (MinIO for on-prem, S3 for cloud). Same container images everywhere.
**Notes:** This reinforces the docker-compose delivery decision (D-006). Architecture must avoid hard cloud dependencies: use S3-compatible (not S3-only), SMTP (not SES-only), etc. Multi-tenant for SaaS, single-tenant for on-prem — same codebase, tenant isolation via config.

### Q13: White-label / Multi-tenant Depth
**Question:** White-label support? (A) Multi-tenant only, single Argus branding, (B) White-label (operator branding/domain), (C) A now, white-label later
**Answer:** A — multi-tenant only, Argus branding.
**Decision:** Multi-tenant with Argus branding. No white-label in v1. Tenant isolation for data/config but single UI brand.
**Notes:** Keeps frontend complexity down. White-label is mostly theming + custom domain — can be added later without architectural changes if tenant model is solid from the start.

### Q14: Scope Gap Analysis — Competitive Parity Check
**Question:** Are we leaving anything critical out of scope that would put us behind global/local competitors?
**Discussion:** Systematic audit against Enea, Alepo, emnify, Mobile ITM SCM. Found 12 potential gaps.
**Answer:** Approved adding all critical (🔴) and important (🟡) items to scope.
**Decision:** Scope expanded with competitive parity additions:
- L1: + 5G SBA proxy (HTTP/2 AUSF/UDM), + Network slice authentication
- L3: + Steering of Roaming (SoR), + OTA SIM management (APDU), + MSISDN number pooling
- L4: + Slice-aware policy rules
- L5: + CDR processing & rating engine, + Carrier cost tracking
- Portal/API: + SMS Gateway (outbound), + Event streaming (WebSocket/SSE)
Explicitly out of scope (low risk): VoWiFi/WiFi Offload, TACACS+, Geo-fencing, Device management (firmware/health)
**Notes:** With these additions, Argus has full competitive parity with global leaders (Enea, Alepo for AAA depth) AND IoT platforms (emnify for connectivity management). No single competitor covers all 5 layers + these additions — that IS the differentiator.

### Final Summary — CONFIRMED
User approved the complete project summary with all 5 layers + competitive parity additions. Discovery phase complete.
