# Bug Patterns & Prevention Rules

Runtime knowledge base of bugs that have occurred and rules to prevent them.
Read by: Planner (warnings), Gate/Scouts (compliance check), Developer (awareness).

## Patterns

- [2026-04-17] PAT-001 [STORY-084]: Double-writer on abort/session-end metric vectors â€” Root Cause: Metric increment lives in both the low-level client (on error return) AND the high-level engine (on classified session outcome); a single logical abort double-counts when both layers fire. â€” Prevention: For every session-level counter (e.g. `*SessionAbortedTotal`, `*SessionFailureTotal`), designate exactly one writer (typically the engine / outermost orchestrator); inner layers return wrapped sentinel errors and do NOT touch counters. Document the invariant in the package's `doc.go`. Tests must assert the TOTAL across reason buckets for one logical abort equals exactly 1. â€” Affected: metrics layer (any protocol client + engine pair).
- [2026-04-17] PAT-002 [STORY-085]: Single-clock polling loop that misses a dynamically-shortened deadline â€” Root Cause: An interim/polling loop driven by a fixed `ticker` (N-second interval) cannot honour a deadline shorter than N seconds away because the loop only wakes on the ticker. When an external event (e.g. CoA-Request) shortens the session deadline to a value < N, the session overshoots by up to N seconds. Discovered as STORY-085 Gate F-A6. â€” Prevention: Use a composite `select` with both the periodic `ticker.C` and a `time.Timer.C` that is armed (or re-armed) to the effective deadline whenever that deadline is updated. The timer fires exactly when the deadline is reached; the ticker continues to drive interims. Stop and drain the timer in the session's `defer` block to prevent goroutine leaks. Covered by `TestSessionTimeout_SubIntervalDeadlineFires` (VAL-027). â€” Affected: any engine or worker with an interim/polling loop whose deadline is externally mutable (CoA, policy update, admin override).
- [2026-04-18] PAT-003 [STORY-090 Gate F-A1]: Metric label set not expanded when write-fanout grows along a new dimension â€” Root Cause: A gauge keyed on `{operator_id}` was written concurrently from N per-protocol goroutines; last writer wins and all-but-one protocol's state was silently discarded. Discovered when AC-10 required per-protocol gauge series. â€” Prevention: Expand the metric label set BEFORE introducing the write-fanout, not after. Add a dimension label (e.g. `protocol`) the same commit that adds per-dimension goroutines/writers. The rename is a breaking change â€” update Grafana dashboard expr + Prometheus alert rule + legend format atomically. â€” Affected: any metric with fan-out writers that produces a `last-write-wins` semantic on a single label-set key. Covered by (VAL-028), `TestRegistry_SetOperatorHealth` + `TestHealthChecker_FansOutPerProtocol`.
- [2026-04-18] PAT-004 [STORY-090 Gate F-A5]: Goroutine cardinality N_parent Ă— M_child per scheduler tick â€” Root Cause: HealthChecker spawned one goroutine per (operator, protocol) per 30-second tick, producing up to 500 goroutines for 100 operators Ă— 5 protocols. Discovered in Gate analysis (F-A5). â€” Prevention: When a parent entity has M children and all M perform periodic work with the same interval, collapse to one ticker per parent that iterates children sequentially inside the goroutine. Only introduce per-child goroutines when child probes are independently timeable or when M Ă— probe_duration â‰Ą interval. Test with `TestHealthChecker_StartOperatorLoop_SingleTickerPerOperator`. â€” Affected: any scheduler that maps over a nested collection (operatorâ†’protocols, tenantâ†’users, poolâ†’addresses). Covered by (VAL-031).
- [2026-04-18] PAT-005 [STORY-090 Gate F-A2]: Masked-secret sentinel persisted via PATCH round-trip â€” Root Cause: GET API emitted `"****"` sentinel for secret fields in `adapter_config`. A subsequent PATCH with the same payload (unchanged secrets) would silently persist `"****"` as the literal stored secret value, breaking all downstream auth. â€” Prevention: Before encrypting and storing a PATCH body, splice the stored plaintext back into any field whose new value equals the sentinel. Additionally, if the stored envelope is undecryptable AND the body carries a sentinel, return 422 (`masked_sentinel_without_stored`) rather than silently persisting garbage. Apply the restore step atomically with validation/encrypt â€” never mutate a secret to `"****"`. â€” Affected: any API that emits a redaction sentinel on GET and accepts the same DTO shape on PATCH/PUT. Covered by (VAL-029), `TestRestoreMaskedSecrets_RestoresSentinelFromStored`.
- [2026-04-20] PAT-006 [FIX-201 Gate F-A1]: Shared payload struct field silently omitted at construction sites â€” Root Cause: A new field (`Reason`) was added to a shared payload struct (`BulkPolicyAssignPayload`, `BulkEsimSwitchPayload`). The handler added it to the `state-change` branch but forgot to populate it in the `policy-assign` and `operator-switch` sim_ids branches. The struct literal compiled without error (Go zero-value default), so the bug was invisible until tests explicitly asserted the field value round-trip. â€” Prevention: When adding a field to a shared payload/DTO struct, immediately grep all handler-side and processor-side struct literal construction sites (`grep -rn 'StructName{'`) and verify every site explicitly sets the new field. Add or extend a regression test that captures the serialized payload and asserts the new field's value equals the source request field (not zero). Do NOT rely on the compiler to catch missing assignments in struct literals. â€” Affected: any Go struct used as a payload/DTO that is constructed in multiple handler branches or packages. Covered by `TestBulkPolicyAssign_SimIdsArray_ReasonPropagatedToPayload` and `TestBulkOperatorSwitch_SimIdsArray_ReasonPropagatedToPayload` (FIX-201 gate).
- [2026-04-20] PAT-007 [FIX-202 Gate F-A1]: Mutex alone does not enforce happens-before ordering between goroutines for a shared map â€” Root Cause: In `internal/api/dashboard/handler.go`, the `active_sessions` merge into the operator-health map was placed inside the operator-health goroutine body, protected by a mutex. However, the session-stats goroutine (which produces `sessionStatsByOp`) ran concurrently, so the merge could execute before `sessionStatsByOp` was populated â€” the mutex prevented a data race on the map object itself but could not prevent the read-before-write logic error (intermittent nil map â†’ silently dropped `active_sessions`). â€” Prevention: In a goroutine graph where goroutine A writes a map and goroutine B must read it, the read MUST happen after a synchronization point that guarantees A has completed (e.g. `wg.Wait()`, channel receive, or structured pipeline). Placing the read inside its own mutex-protected goroutine is insufficient when the writer runs concurrently and may not have completed. Pattern: write in the producing goroutines, synchronize with `wg.Wait()`, read sequentially after the wait. â€” Affected: any concurrent fan-out pattern (parallel goroutines writing to shared maps/slices) where a downstream step requires the combined output of all producers. Covered by `go test -race ./internal/api/dashboard/...` (FIX-202 gate).
- [2026-04-20] PAT-009 [FIX-204]: Nullable FK columns in analytics aggregations â€” always COALESCE before projecting; align sentinel string between time-series and breakdown query branches. â€” Root Cause: `GetTimeSeries` in `internal/store/usage_analytics.go` projected `apn_id::text AS group_key` without a NULL guard. Some CDR rows have `apn_id = NULL` (orphan sessions, fallback APNs). `pgx` panics scanning SQL NULL into a Go `string` field â€” runtime 500. `GetBreakdowns` used a different sentinel `'unknown'` (potentially ambiguous with real rat_type values). These two query branches were independently maintained and diverged. â€” Prevention: (a) Any SQL column that appears in a `SELECT` clause and may be NULL MUST be wrapped with `COALESCE(col::text, '__unassigned__')` before being scanned into a Go `string`. (b) The sentinel string MUST be a single constant used across ALL query branches that contribute to the same API response surface (time-series, breakdowns, totals) to avoid inconsistent strings in the same JSON payload. (c) Add a pure-Go SQL-fragment test (`TestBuildTimeSeriesQuery_COALESCESentinel`) that asserts the COALESCE is present in the assembled SQL string â€” this guards against future refactors stripping the NULL protection. (d) If both a handler-level name resolver and a FE label mapper exist, designate exactly one as the translation point per dimension (for rat_type: FE-only; for apn/operator: handler resolves UUID, FE handles sentinel). â€” Affected: any analytics or reporting query with a nullable group_by dimension. Covered by `TestBuildTimeSeriesQuery_COALESCESentinel` (FIX-204 gate).
- [2026-04-20] PAT-008 [FIX-203]: Threshold-based delta triggers with first-value cold-start â€” use a sentinel (0 or nil) to suppress the trigger on the first tick; only compare when both prev and curr are valid data. Prevents startup noise and divide-by-zero errors on transient timeouts. â€” Root Cause: When a new `lastLatency map[healthKey]int` entry is seeded with 0 on operator registration, the first probe returns a real latency (e.g. 200ms). Without a cold-start guard, `|200-0|/max(0,1) = 200 = 20000%` would always exceed the 10% threshold and fire an event on every first tick â€” flooding NATS on startup. The fix: gate the latency-delta branch with `prevLatency > 0 && currLatency > 0`. Cold start (`prevLatency == 0`) falls through without publishing; the second tick sees two real values and begins comparing. The status-flip path is unaffected (no cold-start issue there). â€” Prevention: For any threshold-based delta trigger (`abs(curr - prev) / prev > threshold%`): (a) initialize the `prev` accumulator to a sentinel value (0 for int, nil for pointer) distinct from any real measurement; (b) guard the delta comparison with `prev > sentinel && curr > sentinel`; (c) add an explicit test case `TestCheckOperator_SuppressesOnColdStart` seeding `prev=0, curr=200` and asserting the event is NOT published. Apply the same pattern to any periodic comparator that uses the first sample as a baseline (rate calculators, SLA breach checks, anomaly detectors). â€” Affected: `internal/operator/health.go` `HealthChecker.checkOperator` latency-delta gate at `:412-437`; any future per-entity delta comparator (auth_rate drift, session-count change alerting). Covered by `TestCheckOperator_SuppressesSubThresholdLatency` + `TestCheckOperator_StatusFlipStillPublishes` (FIX-203 gate, 3278 PASS).

- [2026-04-20] PAT-010 [FIX-205]: **Single-flight pattern for dependency-triggered refresh â€” module-level mutex + failedQueue prevents N parallel refreshes from the same expiry trigger.** â€” Root Cause: A page that fires N concurrent API calls on mount will receive N 401 responses if the token has expired. Without a single-flight guard, each 401 independently triggers a `POST /auth/refresh`, leading to N refresh calls ("refresh storm"), N token rotations, and N-1 stale tokens returned to the other requests â€” breaking them even after a successful refresh. â€” Prevention: Use a module-level (not React-state) `isRefreshing: boolean` flag and a `failedQueue` array in the HTTP client layer. On 401: if `!isRefreshing` â†’ set flag, fire one refresh call; if already `isRefreshing` â†’ push `{resolve, reject}` to queue instead of calling refresh. On refresh settle: drain the queue (resolve or reject all), clear flag. The refresh call MUST use the raw HTTP client (not the intercepted `api` instance) to prevent the response interceptor from re-entering on the refresh response itself. `originalRequest._retry = true` provides a secondary guard for any path where the raw client cannot be guaranteed. â€” Affected: Any FE HTTP client that intercepts 401 and calls a token refresh endpoint; especially relevant when multiple components mount simultaneously and fire API calls (dashboard, list pages with parallel sub-queries). â€” Implementation: `web/src/lib/api.ts` â€” `isRefreshing`, `failedQueue`, `processQueue`.

- [2026-04-21] PAT-013 [FIX-211]: PostgreSQL stores `CHECK (col IN (...))` as `= ANY (ARRAY[...])` in `pg_get_constraintdef` â€” ILIKE `%in%` does NOT match. â€” Root Cause: FIX-211 migration `20260421000003_severity_taxonomy_unification.up.sql` used a `DO` block that looked up the old anomalies severity CHECK with `pg_get_constraintdef(oid) ILIKE '%severity%in%'` and dynamically DROPPED it. PG renders IN-lists as `= ANY(ARRAY[â€¦])`, so the ILIKE returned NULL, the DROP was skipped, and the old 4-value CHECK (`anomalies_severity_check`) remained alongside the new 5-value `chk_anomalies_severity`. Both constraints were ANDed at INSERT time â€” any `info` row would fail the OLD constraint even though the NEW one allowed it. The migration `dirty` flag was false, masking the silent partial success. â€” Prevention: (a) For constraint-drop DO blocks, match on the column reference only (`ILIKE '%severity%'` excluding the new constraint by name via `conname <> 'chk_<new>'`) rather than the SQL-syntax form of the IN-list. (b) When the constraint name is known and deterministic (PG's default pattern is `<table>_<column>_check`), add an explicit `ALTER TABLE â€¦ DROP CONSTRAINT IF EXISTS <name>` before the dynamic fallback. (c) After adding a CHECK that adds values, insert a sentinel row (or a SELECT probe) in the migration to confirm the new values are actually accepted end-to-end. â€” Affected: any migration that drops/replaces a PostgreSQL CHECK constraint via dynamic SQL. Covered by updated migration file that uses both explicit IF EXISTS + a `FOR cons_name IN ... LOOP` over `pg_get_constraintdef ILIKE '%severity%' AND conname <> '<new>'`.
