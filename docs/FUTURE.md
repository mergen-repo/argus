# Future Roadmap — Argus

> Generated by Amil Feature Discovery
> Date: 2026-03-18
> Based on: competitor analysis, industry trends, market research

## Overview

This document outlines potential future features and phases beyond the current MVP scope.
These are NOT committed features — they represent directions the product could evolve.

---

## Future Phase: AI & Predictive Intelligence

**Theme:** AI-driven proactive network management and autonomous decision-making
**Rationale:** In 2026, all major platforms (AT&T, emnify, Spenza) are deploying AI-driven autonomous decision mechanisms. Argus's analytics layer (L5) provides the data foundation — this phase adds ML-powered intelligence on top.
**Inspired by:** AT&T IoT Network Intelligence (CES 2026), emnify SuperNetwork autonomous micro-decisions, Vodafone digital twin optimization, Spenza AI-driven cost optimization

### Features

| ID | Feature | Description | User Benefit | Complexity |
|----|---------|-------------|-------------|------------|
| FTR-001 | AI Anomaly Engine | ML-based anomaly detection beyond rule-based — learns "normal" patterns per SIM/APN, flags meaningful deviations. Detects SIM cloning, fraud, DDoS before impact. | Proactive threat detection instead of reactive alerts | XL |
| FTR-002 | Predictive Quota Management | Per-SIM/APN quota consumption forecast ("this SIM will exhaust quota in 3 days"). Auto-actions: suggest upgrade, throttle, notify. | Prevent service disruption, optimize costs | L |
| FTR-003 | Auto-SoR (Autonomous Steering) | AI replaces manual SoR rules. Real-time operator selection based on coverage + cost + latency optimization per device location. | Lower costs, better coverage without manual intervention | XL |
| FTR-004 | Network Quality Scoring | ML-based quality score per operator/APN/region with trend analysis. Historical and predictive quality ratings. | Data-driven operator selection and SLA negotiation | M |

### Architecture Implications
- L5 Analytics must expose raw data pipeline for ML model training
- Event streaming (NATS) must support ML model inference triggers
- Policy engine (L4) must accept AI-generated policy recommendations
- SoR engine (L3) must support pluggable decision strategies (rule-based → AI-based)
- TimescaleDB time-series data is ideal training data source

## Future Phase: Digital Twin & Network Simulation

**Theme:** Virtual replica of the entire SIM fleet for risk-free testing and optimization
**Rationale:** Vodafone and AT&T achieved 20-40% operational cost savings using digital twins. With 10M SIMs, testing changes in production is extremely risky — a digital twin enables simulation before deployment.
**Inspired by:** Vodafone 10K site digital twin, IETF Network Digital Twin Architecture (draft-irtf-nmrg-network-digital-twin-arch), Deutsche Telekom AI-driven optimization, NVIDIA 6G digital twin tools

### Features

| ID | Feature | Description | User Benefit | Complexity |
|----|---------|-------------|-------------|------------|
| FTR-005 | Network Digital Twin | Virtual replica of entire SIM fleet. Simulate policy changes, operator switches, failover scenarios and see projected impact before applying to production. | Zero-risk testing of changes affecting millions of SIMs | XL |
| FTR-006 | What-If Scenarios | Interactive scenario builder: "What if we drop Turkcell and run Vodafone+TT only?" — simulates cost, coverage, latency impact with real historical data. | Data-driven strategic decisions for operator portfolio | L |
| FTR-007 | Load Testing Simulator | Simulate 10M concurrent auth requests against AAA core. Identify bottlenecks, capacity limits, breaking points before they happen in production. | Capacity planning, performance assurance | M |

### Architecture Implications
- AAA core must support "simulation mode" (process requests without real operator calls)
- Policy engine must support "shadow mode" (evaluate rules without enforcement)
- Historical data (TimescaleDB) must be queryable for scenario simulation
- Operator mock simulator (already in MVP) can be extended for digital twin
- Need separate compute resources for simulation workloads (don't impact production)
- STORY-089 (operator-SoR simulator, `cmd/operator-sim` binary + `argus-operator-sim` container) is the long-term home for the minimal `Nsmf_PDUSession` mock shipped by STORY-092 at `internal/aaa/sba/nsmf.go`. Either absorb the handler verbatim into the operator-sim binary or retire the in-process mock once adapter config can point at an external operator-sim container. STORY-092 kept interfaces (`SIMResolver`/`SIMUpdater`/`IPPoolOperations`/`SIMCache`) stable to preserve either absorption path.

## Observability Extension Points

These items are architectural scope boundaries from STORY-065 — the underlying APIs do not yet exist. They are not deferrals; they are follow-up work pending sibling API evolution.

| Item | Description | Blocked By | Decision |
|------|-------------|-----------|----------|
| NATS Pending-Messages Poller | Wire `argus_nats_pending_messages` Gauge to a real consumer lag count. Currently stays 0 because `EventBus` exposes no `PendingByConsumer` API. Requires adding a consumer introspection method to `internal/bus/nats.go`. | `EventBus.PendingByConsumer` API (not yet implemented) | DEV-174 |
| Diameter/SBA Prometheus Recorders | Wire Prometheus `PrometheusRecorder` into Diameter and 5G SBA AAA servers (analogous to RADIUS wiring in STORY-065). Currently blocked because Diameter and SBA servers expose no `SetMetricsRecorder` method. | `diameter.Server.SetMetricsRecorder` + `sba.Server.SetMetricsRecorder` (not yet implemented) | DEV-175 |
| `METRICS_TENANT_LABEL_ENABLED` Active Enforcement | The env var is defined and validated but currently passive — it does not dynamically suppress tenant_id labels at runtime. Activating it requires a registry rebuild or label-conditional recording path. Reserved as a cardinality kill-switch for production emergencies. | Implementation complexity, no current cardinality pressure | DEV-173 |
