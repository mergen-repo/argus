## 1. Next Story Impact Analysis

### STORY-036 (Anomaly Detection Engine) — 1 observation

1. **Data spike detection needs per-SIM cost queries:** STORY-036 AC requires "SIM daily usage exceeds 3x its 30-day average." STORY-035's `GetLowUsageSIMs` and `GetTopExpensiveSIMs` query patterns (GROUP BY sim_id on raw `cdrs` table) demonstrate the exact query approach STORY-036 will need for per-SIM anomaly thresholds. The `buildConditions()` helper in `cost_analytics.go` is specific to cost queries (no reuse needed), but the pattern is established.

2. **Suggestion engine as anomaly source:** The cost optimization suggestions (inactive SIMs, operator switch candidates) could serve as non-security anomaly inputs in a future enhancement. Not blocking, but worth noting that STORY-036 anomaly types are security-focused (cloning, floods, spikes) and cost anomalies remain in SVC-07 cost module.

**Impact: NONE — No blocking changes needed.**

### STORY-037 (Connectivity Diagnostics) — No changes needed

STORY-037 is a per-SIM diagnostic using session, operator, APN, and policy data. It does not depend on cost analytics. No impact.

**Impact: NONE.**

### STORY-048 (Frontend Analytics Pages) — 2 observations

1. **API response shape confirmed:** STORY-048 AC specifies "cost cards, carrier comparison chart, optimization suggestions panel." The `CostAnalytics` response DTO has all required fields: `total_cost`, `currency`, `by_operator`, `cost_per_mb`, `top_expensive_sims`, `trend`, `comparison`, `suggestions`. The `Suggestion` struct includes `description`, `affected_sim_count`, `potential_savings`, `action` -- all matching STORY-048's "optimization suggestions panel with cards" requirement.

2. **Action button integration:** STORY-048 AC says "action button links to bulk operation (e.g., Switch 500 SIMs -> bulk operator switch)." The `Suggestion.Action` values (`operator_switch`, `terminate`, `plan_downgrade`) map to existing bulk operations from STORY-030 (`bulk_esim_switch`, `bulk_state_change`). Frontend will need to translate suggestion actions to bulk operation API calls.

**Impact: LOW — No API changes needed. Frontend mapping is straightforward.**

### STORY-053 (Data Volume Optimization) — 1 observation

1. **Cost queries use raw `cdrs` table:** 6 of 8 store methods query the raw `cdrs` table directly (not continuous aggregates). Only `GetCostTrend` uses `cdrs_daily`. If STORY-053 compresses or archives old CDR data aggressively, cost analytics for historical periods will lose granularity. STORY-053 should ensure that `cdrs_daily` and `cdrs_hourly` aggregates are preserved even after raw CDR compression (TimescaleDB compression preserves continuous aggregate data by design).

**Impact: LOW — TimescaleDB compression handles this correctly by design.**

## 3. New Domain Terms

| Term | Definition | Context |
|------|-----------|---------|
| Cost Analytics | Fleet-wide cost analysis API providing total cost, per-carrier breakdown, cost per MB by operator/RAT, top 20 expensive SIMs, cost trend, period comparison, and automated optimization suggestions.
